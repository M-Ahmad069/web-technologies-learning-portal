Categories of Web Applications and Web Architectures

Lecture 2 • Reading time: 10 minutes

Web applications use different structures and architectures to work efficiently and scale. This lecture covers how we categorize web applications by purpose and structure, and how architectural patterns organize system components for maintainability and growth.

Introduction

Web applications are software programs that run in the browser and interact with servers. Depending on their purpose—document editing, social interaction, or data discovery—they are built with different architectures. Choosing the right category and architecture helps applications perform well and scale.

Why it matters: Different architectures suit different needs. Document-centric apps prioritize collaboration; social apps prioritize real-time interaction; semantic apps prioritize structured data. Understanding these categories helps you design and build the right solution.

Categories of Web Applications

Web applications can be categorized based on their purpose and structure. The three main categories are:

Document-Centric Web

Applications focused on delivering and editing documents and information. Users create, share, and collaborate on content.

Examples: Google Docs, Microsoft Office Online, Dropbox Paper

Social Web Applications

Platforms that enable user interaction, content sharing, and community building.

Examples: Facebook, Twitter, Instagram, LinkedIn

Semantic Web Applications

Applications that use structured, machine-readable data for enhanced search and data integration.

Examples: Wolfram Alpha, Google Knowledge Graph, Schema.org

Document-Centric Web Applications

Document-centric applications focus on delivering documents and information. They emphasize creation, editing, sharing, and versioning of content rather than social interaction.

Examples: Google Docs, Microsoft Office Online, Dropbox Paper

Key features

  • Real-time collaboration
  • Version control
  • Document sharing
  • Editing tools

Social Web Applications

Social web applications enable user interaction and content sharing. They center on profiles, feeds, and communication between users.

Examples: Facebook, Twitter, Instagram, LinkedIn

Key features

  • User profiles
  • News feeds
  • Likes and comments
  • Notifications

Semantic Web Applications

Semantic web applications use structured data and machine-readable information. They improve search, data integration, and context-aware retrieval by linking and organizing data in meaningful ways.

Examples: Wolfram Alpha, Google Knowledge Graph, Schema.org

Key features

  • Enhanced search capabilities
  • Data integration
  • Context-aware information retrieval
  • Improved data relationships

Interoperability

Definition

Interoperability is the ability of different systems or applications to work together and exchange data effectively. In web applications, it allows services from different providers to communicate using common standards and protocols.

Introduction to Web Architectures

Web architectures define how system components interact with each other. They determine where logic runs, how data flows, and how the system can scale. Common patterns include:

  • Layered Architecture — Organizes code into horizontal layers (e.g. presentation, business logic, data).
  • One-Tier Architecture — All components in a single system (e.g. desktop app).
  • Two-Tier Architecture — Client and server (e.g. classic client-server).
  • Three-Tier Architecture — Presentation, application, and data tiers.
  • N-Layered Architecture — Multiple layers for flexibility and modularity.

Layered Architecture

In layered architecture, the system is divided into horizontal layers. Each layer has a distinct responsibility and communicates with adjacent layers.

Presentation Layer
Business Logic Layer
Data Layer

Benefits:

  • Separation of concerns
  • Modularity
  • Scalability

One-Tier Architecture

In one-tier architecture, all components exist in a single system. The application, data, and user interface run on one machine with no separate client or server.

Example: Desktop applications like Microsoft Excel, where the program and data reside on the same computer.

Two-Tier Architecture

Two-tier (client-server) architecture splits the system into two parts: the client handles the user interface and user input, while the server handles the database and business logic.

Example: Simple client-server database applications where a desktop or web client talks directly to a database server.

Three-Tier Architecture

Three-tier architecture separates the system into three tiers: Presentation (UI), Application (business logic), and Data (database). This is common in modern web applications.

Presentation Tier
Application Tier
Data Tier

Benefits:

  • Scalability
  • Easier maintenance
  • Separation of concerns

N-Layered Architecture

N-layered architecture extends the layered model by adding more layers. Each layer has a clear responsibility, and dependencies flow in one direction (e.g. presentation → application → domain → data access).

Examples of layers: Presentation, Application, Domain, Data Access

Benefits:

  • Flexibility
  • Modular architecture
  • High scalability

Real World Examples

Different architectures in practice:

Amazon

Large-scale e-commerce and cloud platform using multi-tier and distributed architecture.

Enterprise banking systems

N-layered and three-tier architectures for security, compliance, and scalability.

Client-server database apps

Two-tier architecture with a client UI and a database server.

ERP systems

N-layered architecture for integration of business processes and data.

Lecture Summary

Key Takeaways

  • Web applications can be categorized into document-centric, social web, and semantic web.
  • Different architectures organize how systems are built (layered, one-tier, two-tier, three-tier, N-layered).
  • Layered architectures improve scalability and modularity.
  • Multi-tier architectures are common in modern web applications.
